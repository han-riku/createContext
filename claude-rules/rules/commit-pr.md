# コミット・PR規約（詳細）

## コミットメッセージ形式

Conventional Commitsに従う。

```
<type>: <emoji> <説明>
```

- 例: `feat: ✨ 決済APIを追加`
- 例: `fix: 🐛 タイムアウト処理を修正`
- 破壊的変更は `!` を付与（例: `feat!: ✨ APIレスポンス形式を変更`）

## タイプ一覧

| type | 絵文字 | 用途 | 例 |
|---|---|---|---|
| `feat` | ✨ | 新規機能（ファイル）の追加 | `feat: ✨ faviconファイルの追加、及び、設定` |
| `fix` | 🐛 | バグ修正 | `fix: 🐛 未登録のメールアドレスが入力された場合に500エラーとなる箇所を改修` |
| `docs` | - | ドキュメントのみの変更 | `docs: READMEにセットアップ手順を追加` |
| `refactor` | ♻️ | リファクタリング（同じ挙動を担保した上での変更）・ファイル名変更やディレクトリ変更などソース変更を伴わない変更も含む | `refactor: ♻️ n+1問題が発生していたループをeager_loadに変更` |
| `test` | ✅ | テストの追加 | `test: ✅ Userモデルのテストを追加` |
| `chore` | - | ビルド・依存関係・設定など機能に直接関わらない雑務 | `chore: 依存パッケージを更新` |

補助的な変更種別（`type` ではなく絵文字のみで表現する場合）:

| 絵文字 | 用途 | 例 |
|---|---|---|
| 👍 | 機能修正（バグではない） | `fix: 👍 nameのプレースホルダを「氏名」から「お名前」に変更` |
| 🚿 | 不要な機能（ファイル）の削除 | `chore: 🚿 未使用の関数「set_name」を削除` |

- 絵文字はコミットメッセージへの必須付与項目とする（分類のため）
- 上記表に当てはまらない変更は最も近いtype/絵文字を選択し、判断に迷う場合はHuman-Reviewへ

## PR規約

- **1PR1目的**: 複数の独立した変更を1つのPRにまとめない
- **PRタイトル**: コミットメッセージと同様にConventional Commits形式（絵文字含む）で記述
- **チケット紐付け**: PRタイトルにBacklogのチケット番号を必ず含める（例: `feat(auth): ✨ PROJECT-123 ログイン機能を追加`）・チケット番号のないPRは原則マージ禁止
- **PRの説明**: 破壊的変更・仕様変更・影響範囲の大きい重要な更新がある場合は、その内容と影響範囲をPRの説明に必ず明記する
- **直接push禁止**: `main` / `develop` ブランチへの直接pushを禁止・必ずPR経由でマージ
